# Отчет по проекту Food Journal Tracker

## 1. Что это за работа
В рамках задания мне нужно было разобрать существующий проект на Expo / React Native, найти проблемы, связанные в том числе с использованием `expo-sqlite`, исправить их и привести приложение в рабочее состояние. Дополнительно требовалось улучшить интерфейс и подготовить документацию по проделанной работе.

## 2. Как я подошел к задаче
Я начал с просмотра структуры проекта и зависимостей, после чего отдельно проверил:

- точку входа приложения
- экран авторизации
- экран журнала
- слой работы с базой данных
- конфигурацию Expo и зависимости, влияющие на web-запуск

Дальше я сверил реализацию `expo-sqlite` с актуальной документацией Expo, поднял проект локально, проверил web-сборку и начал исправлять найденные проблемы по очереди.

## 3. Что было не так в исходном состоянии
По ходу работы я выявил несколько проблем:

- слой базы данных был привязан к устаревшим представлениям о формате SQL-результатов
- часть логики не соответствовала текущему async API `expo-sqlite`
- web-версия проекта запускалась нестабильно
- экран журнала содержал платформенно-зависимые части, которые плохо вели себя в браузере
- в проекте были несовпадения по версиям зависимостей относительно Expo SDK 52
- не хватало понятного тестового сценария для быстрой проверки приложения

## 4. Основные исправления

### 4.1. Слой хранения данных
Я переработал модуль базы данных и привел его к актуальному async API `expo-sqlite`.

Что было сделано:
- инициализация базы переведена на `openDatabaseAsync`
- запросы приведены к `getFirstAsync`, `getAllAsync`, `runAsync`
- миграции оформлены через `withExclusiveTransactionAsync`
- добавлен контроль версии схемы через `PRAGMA user_version`
- добавлены вспомогательные функции `initDatabase`, `getDatabase`, `getFirst`, `getAll`, `run`

Отдельно для браузера я сделал файл `components/database/database.web.js`, в котором реализовал хранение данных через `localStorage`. Это решение позволило избежать проблем, связанных с web-версией SQLite в Expo, и сделать браузерную версию стабильной.

### 4.2. Авторизация
На экране авторизации я пересобрал логику входа и регистрации так, чтобы она работала с новым форматом ответов от базы.

Что изменилось:
- запросы к пользователям переписаны под новый слой данных
- добавлена нормализация email через `trim().toLowerCase()`
- улучшена базовая валидация формы
- после успешной регистрации и входа пользователь сразу попадает на экран журнала

### 4.3. Экран журнала
Экран журнала я переработал сильнее всего, потому что именно там было больше всего поводов для ошибок.

Что исправил:
- загрузку записей из хранилища
- создание новой записи
- обновление существующей записи
- удаление записи
- фильтрацию по категориям

Что дополнительно улучшил:
- добавил поле настроения после еды
- добавил блок со статистикой
- добавил кнопку выхода из аккаунта
- подготовил отдельную web-реализацию экрана `screens/homeScreen.web.js`, чтобы убрать конфликты браузерной версии с камерой, свайпами и вложенными списками

## 5. Проблемы запуска и как они были решены

### Проблема 1. `npm` и `npx` не запускались из PowerShell
Причина была в локальной execution policy. В качестве обхода я использовал `npm.cmd` и `npx.cmd`.

### Проблема 2. Web-запуск был нестабильным
После проверки зависимостей выяснилось, что часть пакетов нужно выровнять под Expo SDK 52. Я обновил и скорректировал версии зависимостей, а также добавил отсутствующий `react-native-gesture-handler`.

### Проблема 3. Белый экран в web-версии
На web-версии проект вел себя нестабильно после авторизации. Чтобы убрать эту проблему, я разделил поведение экрана журнала:

- `homeScreen.js` остался для native
- `homeScreen.web.js` сделан отдельно для браузера

В web-версии я убрал все лишние платформенные сложности и оставил простой, предсказуемый интерфейс со скроллом, выбором изображения, созданием и редактированием записей.

## 6. Что я добавил для удобной проверки
Чтобы приложение можно было быстро открыть и проверить без ручной подготовки данных, я добавил демо-аккаунт:

Также я заполнил его несколькими демонстрационными записями, чтобы после входа сразу был виден рабочий журнал.
